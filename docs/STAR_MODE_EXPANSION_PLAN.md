# classpage 별점모드 확장 설계

이 문서는 현재 MVP 이후에 별점모드를 어떻게 확장하는 것이 안전한지 정리한 설계 문서입니다.

핵심 원칙은 그대로 유지합니다.

- 수집과 집계는 바깥에서 한다.
- classpage는 읽고 보여주는 쪽에 머문다.
- 학생 공개 점수와 교사 내부 조정은 계속 분리한다.

## 1. 현재 구조 요약

현재 별점모드는 아래 구조입니다.

1. 학급용/수업용 Google Form 응답이 Google Sheets에 쌓입니다.
2. Apps Script가 `Config.gs`의 규칙 정의를 읽어 자동 적립 규칙을 적용합니다.
3. `별점 수동 조정` 시트 행도 함께 읽습니다.
4. `star-ledger.json`을 생성합니다.
5. classpage는 교사용 화면에서 이 ledger를 읽기만 합니다.

현재 classpage 안에서는 점수 규칙을 직접 저장하거나, 외부 시트에 직접 쓰지 않습니다.

## 2. 무엇을 어디에 두는 것이 맞는가

### 규칙 편집

권장 위치:

- 1차 권장: Apps Script `Config.gs`
- 이유: 자동 적립 규칙과 수동 조정 규칙은 집계 레이어의 기준값이기 때문입니다.

여기에서 관리할 항목:

- `ruleId`
- `label`
- `category`
- `delta`
- `visibility`
- `enabled`
- `sources`
- `allowCustomDelta`
- `autoCriteria`
- `description`

classpage 설정창은 로컬 Obsidian 설정입니다.
따라서 점수 규칙 같은 운영 기준값을 classpage 설정에 넣으면 아래 문제가 생깁니다.

- 교실 전체의 기준값이 아니라 로컬 UI 설정처럼 보일 수 있음
- 실제 JSON 생성 기준과 plugin 표시 기준이 어긋날 수 있음
- 여러 기기/여러 볼트에서 기준값 동기화가 어려움

즉, 규칙 편집의 진짜 소스 오브 트루스는 여전히 Apps Script 쪽이 맞습니다.

### 점수 / 활성 여부 / 공개 여부 저장 위치

권장 구조:

- 자동 규칙 정의: `Config.gs.rules.starMode.rules`
- 수동 조정 이벤트: Google Sheets `별점 수동 조정` 시트
- 최종 읽기 결과: `star-ledger.json`

이렇게 나누면 역할이 명확합니다.

- 규칙 정의는 코드/설정
- 실제 발생 이벤트는 시트 행
- 표시용 결과는 JSON

자동 규칙은 특히 구조화 입력으로만 판정하는 편이 안전합니다.

- `복습 계획에서 정규화한 복습/수행 상태`
- `문제 맞은 개수`
- `틀린 개수`

반대로 자유서술 메모, 칭찬 문장, 교사 해석이 필요한 값은 자동 적립 기준으로 넣지 않는 편이 좋습니다.

## 3. 수동 부여와 일괄 부여는 어디가 적절한가

### 이번 구조에서 가장 적절한 기본안

권장 위치:

- Google Sheets 기반 `별점 수동 조정` 시트

이유:

- 현재 아키텍처가 이미 Apps Script -> JSON 흐름으로 굳어져 있음
- 교사 여러 명이 함께 보거나 수정할 가능성에 더 잘 맞음
- 집계 이력과 감사 흔적을 행 단위로 남기기 쉬움
- plugin을 쓰기 가능한 앱으로 바꾸지 않아도 됨

### 일괄 부여는 어떻게 할까

권장 방식:

1. `별점 일괄 부여` 시트에서 여러 학생 행을 준비
2. Apps Script가 대기 행을 `별점 수동 조정` 시트의 이벤트 행으로 복사
3. ledger는 최종적으로 `별점 수동 조정` 이벤트만 읽음

권장 최소 스키마:

- `timestamp`
- `ruleId`
- `studentKey` 또는 `email/classroom/number/name`

권장 보조 열:

- `delta`
- `visibility`
- `note`
- `teacher`
- `batchId`

현재 구현에서는 `studentKey`, `teacher`, `batchId`를 바로 읽을 수 있고,
`allowCustomDelta: true`인 규칙만 행별 `delta` 덮어쓰기를 허용합니다.

`별점 일괄 부여` 시트는 staging 용도입니다.
즉, 여러 학생에게 같은 규칙을 한 번에 준비하는 곳이고,
실제 source of truth 이벤트 로그는 여전히 `별점 수동 조정` 시트에 남깁니다.

### plugin UI는 어디까지 하는 것이 맞는가

현재 단계 추천:

- plugin UI는 별점 결과 조회와 운영 확인에 집중
- 직접 쓰기 기능은 넣지 않음

이유:

- 현재 plugin에는 Google 인증/쓰기 경로가 없음
- 쓰기 기능을 넣으면 동기화, 충돌, 권한 문제가 바로 따라옴
- release 직전 단계에서 넣기엔 리스크가 큼

## 4. 나중에 plugin UI를 붙인다면

plugin UI를 완전히 배제할 필요는 없습니다.
다만 역할을 잘라서 붙이는 편이 안전합니다.

### 가능한 1단계

- 규칙 편집 UI가 아니라 규칙 보기 UI
- 현재 활성 규칙, 점수, 공개 범위, 설명 확인
- 규칙 변경은 여전히 Apps Script에서 수행

### 가능한 2단계

- batch draft 생성 UI
- 교사가 학생 여러 명을 선택하고, 규칙/점수/메모를 고르면
  classpage가 Google Sheets에 직접 쓰는 대신
  복붙 가능한 TSV 또는 JSON 초안을 만들어줌

이 경우에도 최종 반영은 운영자가 시트나 Apps Script 입력 경로로 넘기는 편이 안전합니다.

### 권장하지 않는 즉시 구현

- plugin에서 Google Sheets 직접 쓰기
- plugin에서 점수 규칙을 단독 저장
- plugin 내부 로컬 파일만 바꿔 점수 기준을 운영하는 구조

이 방식은 현재 제품 원칙과 충돌합니다.

## 5. 자동 규칙과 수동 규칙을 함께 운영하는 방법

권장 분리:

- 자동 규칙
  학급용/수업용 Form 응답을 기준으로 Apps Script가 생성
- 수동 규칙
  교사 시트 입력으로 행 단위 이벤트 생성

둘 다 최종적으로는 같은 `StarEvent` 구조로 합칩니다.

즉, 합치는 위치는 classpage가 아니라 Apps Script 입니다.

현재 JSON에서는 아래 정도가 함께 내려옵니다.

- 이벤트별 `source`
- 이벤트별 `actor`, `batchId`
- ledger 단위 `sourceSummary`
- 규칙별 `autoCriteria`

장점:

- classpage는 계속 결과만 읽으면 됨
- 자동/수동 이벤트가 같은 ledger 안에서 정렬됨
- 감사와 수정 이력이 `별점 수동 조정` 시트 행으로 남음

운영자가 규칙을 늘릴 때는 아래를 먼저 확인하는 편이 안전합니다.

1. 원본 시트에 구조화 헤더가 이미 있는가
2. 같은 학생/같은 수업 그룹에서 중복 적립되지 않는가
3. 학생 식별이 안 되는 행을 자동으로 제외해도 운영상 문제없는가
4. 학생 공개 규칙인지 교사 전용 규칙인지 명확한가

## 6. 추천 구현 순서

현실적인 다음 순서는 아래입니다.

1. Apps Script `Config.gs`의 별점 규칙을 안정화
2. `별점 수동 조정` 시트 스키마를 문서화
3. 자동 적립 + 수동 조정을 같은 이벤트 모델로 유지
4. classpage 교사용 화면에 규칙/최근 이벤트/학생별 누적 확인 강화
5. 필요하면 나중에 batch draft 보조 UI 검토

## 7. 이번 단계 결론

- 규칙 편집은 Apps Script `Config.gs`가 맞습니다.
- 점수/활성 여부/공개 여부/허용 입력 경로도 Apps Script 규칙 구조에서 관리하는 편이 맞습니다.
- 수동 일괄 부여는 plugin 직접 쓰기보다 Google Sheets 입력 경로가 더 적절합니다.
- classpage는 우선 drill-down과 운영 대시보드 역할에 집중하는 편이 안전합니다.
