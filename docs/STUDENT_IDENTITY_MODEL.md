# classpage 학생 식별 모델

## 1. 목적

학생이 Google Form을 제출할 때 매번 반, 번호, 이름을 직접 쓰지 않아도, 가능한 범위에서는 자동으로 학생을 식별하고 집계 품질을 높이기 위한 모델입니다.

핵심 아이디어는 다음과 같습니다.

- Google 로그인 + 이메일 수집을 켠다.
- 응답 이메일을 기본 식별 키로 쓴다.
- 필요하면 반/번호/이름만 allowlist에서 보완한다.
- 교사용 화면과 별점모드는 이 보완된 학생 정보를 사용한다.

## 2. 현재 구조와 잘 맞는 이유

현재 classpage 구조는 이미 아래 흐름을 전제로 합니다.

`Google Form -> Google Sheets -> 이메일 기준 식별 -> Apps Script -> JSON -> classpage`

즉, 학생 식별도 classpage UI 안에서 하지 않고 Apps Script 집계 단계에서 해결하는 편이 맞습니다.

## 3. 기본 식별 키

기본 식별 키는 이메일입니다.

### 권장 우선순위

1. `email-sha256|이메일기반불투명키`
2. `반|번호|이름`
3. 그래도 없으면 `학생 미확인`

왜 이메일을 우선하는가:

- 이름 오타보다 안정적입니다.
- 같은 이름 학생 구분에 유리합니다.
- 폼마다 반/번호/이름 질문이 달라도 식별 기준을 유지할 수 있습니다.

## 4. Google Form 설정 권장값

### 필수

1. Google Workspace 로그인 필요
2. 이메일 주소 수집
3. 1회 제출 제한(가능하면)

### 권장

- 학교 계정만 허용
- 외부 사용자 차단

## 5. 선택형 allowlist의 역할

allowlist는 기본 필수 단계가 아니라, 필요할 때만 쓰는 보조 기준표입니다.

권장 헤더:

- 이메일 주소
- 반
- 번호
- 이름
- 허가
- 표시이름(선택)
- 별점활성(선택)

즉, allowlist는 아래 두 역할을 동시에 합니다.

1. 이 이메일 학생의 반/번호/이름 기준값 제공
2. 필요한 경우에만 추가 제외 규칙 보조

## 6. 반/번호/이름 자동 보완 규칙

응답 시트에 반/번호/이름이 없더라도, 이메일이 allowlist에 있으면 아래처럼 보완합니다.

### 규칙

- 응답 이메일이 allowlist와 일치하면
  - `classroom` = allowlist의 반
  - `number` = allowlist의 번호
  - `name` = allowlist의 이름
- 응답 시트에 값이 있더라도 allowlist 값을 우선 기준으로 사용할 수 있습니다.
- 교사가 원하면 나중에 "응답값 우선 / allowlist 우선" 옵션을 둘 수 있습니다.

초기 단계 추천은:

> **allowlist 우선**

이유:

- 학생이 이름을 장난스럽게 적는 경우를 줄일 수 있습니다.
- 번호 누락, 반 입력 누락을 막을 수 있습니다.
- 교사용 결과와 별점 누적 키가 안정됩니다.

## 7. 응답 시트 질문 구성 권장

### 최소형

- 이메일 수집(구글 폼 설정)
- 나머지 학습/정서 질문만 받기

이 경우 반/번호/이름 질문을 아예 제거할 수 있습니다.

### 절충형

- 이메일 수집
- 이름만 선택형 또는 짧은 확인용으로 남김

이 경우 표시용 확인은 가능하지만 집계 기준은 이메일로 유지합니다.

## 8. 현재 가능한 자동 인식 수준

### 가능한 것

- Google 로그인 계정 이메일 수집
- 이메일 기반 allowlist 대조
- allowlist 기반 반/번호/이름 자동 보완
- 응답 시트의 `반`, `번호`, `이름` 열이 없어도 집계 가능
- 학생별 중복 응답 제거 시 이메일 기반 dedupe
- 별점모드 studentKey 안정화

### 아직 어려운 것

- Google Form이 로그인 계정의 실명/학번을 자동으로 반영해 질문을 자동 채우는 것
- 구글 계정 프로필 이름만으로 반/번호를 알아내는 것
- 학교 정보시스템과 자동 동기화하는 것

즉,

> **이메일 자동 인식은 가능하지만, 반/번호/이름 자동 완성은 allowlist 같은 기준표가 있어야 현실적으로 안정적입니다.**

## 9. 예외 처리

### 1) 이메일은 있는데 allowlist에 없음

처리:

- allowlist를 끈 기본 상태에서는 이메일 기준으로 계속 집계
- allowlist를 켠 경우에만 `excludedResponseCount` 증가 및 별점 적립 제외 가능

### 2) allowlist에는 있는데 응답 시트 값이 비어 있음

처리:

- allowlist 값으로 자동 보완

### 3) 응답 시트 이름과 allowlist 이름이 다름

처리:

- MVP에서는 allowlist 값을 우선
- 필요하면 나중에 mismatch 로그를 남김

### 4) 학생이 계정을 바꿈

처리:

- allowlist 이메일 수정 필요
- 이전 이메일과의 연속성은 운영자가 관리

## 10. 별점모드와의 연결

별점모드는 studentKey가 흔들리면 안 됩니다.

따라서 추천 규칙은 아래입니다.

- studentKey는 이메일 기반 불투명 키
- 표시 이름은 allowlist의 반/번호/이름 기반
- 감점/가점 누적도 같은 키를 사용

예:

- `studentKey = email-sha256|9f4c2d6a1b7e3c5d9a2f4b6c`
- 표시: `3반 12번 학생A`

## 11. Codex 구현 요구사항

Codex가 구현할 때 확인할 것:

1. 응답 시트 필수 헤더에서 `classroom`, `number`, `name` 의존을 줄일 수 있는지
2. allowlist 매핑 시 반/번호/이름 자동 덮어쓰기가 안정적으로 적용되는지
3. `buildStudentKey_()`가 이메일 우선인지
4. `buildStudentReference_()`가 빈 값일 때 allowlist 보완을 쓰는지
5. 교사용 UI에 `학생 미확인` fallback이 과도하게 보이지 않는지
6. 별점 이벤트도 같은 studentKey 규칙을 쓰는지

## 12. 추천 결론

현재 운영 방향에서는 아래가 가장 적절합니다.

- Google 로그인 + 이메일 수집 유지
- allowlist는 필요할 때만 추가
- 반/번호/이름 문항은 단계적으로 축소 가능
- 집계와 별점은 이메일 기반 불투명 식별 키로 통일
- 표시용 학생 정보는 응답값 우선, 필요하면 allowlist로 보완

이렇게 하면 학생 입력 부담은 줄이고, 집계와 별점의 안정성은 높일 수 있습니다.
